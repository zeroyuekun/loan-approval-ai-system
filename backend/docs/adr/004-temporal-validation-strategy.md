# ADR 004: Temporal Validation Strategy

## Status

Accepted

## Date

2026-03-23

## Context

The model validation strategy must reflect real deployment conditions. In lending, economic conditions change over time — RBA cash rate movements, property market cycles, employment trends, and regulatory changes all affect default rates. The validation approach must produce performance estimates that accurately predict how the model will perform on future, unseen data.

## Decision

Use out-of-time (OOT) validation as the primary model evaluation strategy:

- **Training set (75%):** Earliest quarters of data
- **Validation set (12.5%):** Next quarters after training period (used for hyperparameter tuning and threshold calibration)
- **Test set (12.5%):** Latest quarters (held out entirely until final evaluation)

The temporal split is based on the `quarter` column generated by the synthetic data pipeline (see ADR 001). If the data lacks a temporal column, the system falls back to random stratified splitting with the same proportions.

### Why not random k-fold cross-validation?

In lending data, random splits leak future information into training. Consider a model trained on a random split that includes 2025Q4 data in training and 2024Q1 data in testing. The model has effectively "seen the future" — it knows about rate rises, property corrections, and employment shifts that occurred after Q1 2024. This produces inflated performance metrics that do not reflect true forward-looking accuracy.

Specific failure modes of random splitting in temporal data:

- **Concept drift masking** — model appears stable because train and test share the same economic regime
- **Overfitting to regime** — model memorizes features of a specific rate environment rather than learning generalizable patterns
- **Misleading Gini/AUC** — metrics can be 5-10 points higher than actual production performance

Out-of-time validation reveals these issues because the test set is strictly in the future relative to training, matching how the model will actually be deployed.

## Consequences

**Positive:**

- Realistic performance estimates that match production deployment conditions
- Catches temporal drift and concept shift between training and serving periods
- Matches how the model will be used: trained on historical data, predicting future applications
- Validation set enables proper threshold calibration without test set contamination

**Negative:**

- May show lower metrics than random split (this is the correct behavior — inflated metrics from random splits are misleading)
- Requires temporal metadata in the dataset (quarter column)
- Smaller effective training set compared to k-fold (no data reuse across folds)
- Fallback to random split loses temporal benefits when quarter column is unavailable

## Alternatives Considered

| Alternative | Reason for rejection |
|---|---|
| Random k-fold cross-validation | Leaks future information, inflates metrics, masks temporal drift |
| Expanding window (walk-forward) | More rigorous but computationally expensive for initial development; suitable for production retraining pipeline |
| Time-series cross-validation | Better than random k-fold but still allows some future leakage in early folds |
| Single random train/test split | Same future leakage as k-fold, plus high variance from single split |
