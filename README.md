# Loan Approval AI System

A full-stack loan approval system that processes applications from submission to decision email. ML scores the applicant, Claude writes the correspondence, and an agent pipeline checks everything for bias before anything gets sent. Built around Australian lending regulations — APRA serviceability buffers, NCCP Act responsible lending, Banking Code of Practice disclosure requirements, the lot. The compliance layer is where most of the engineering work went.

## What this is

Three levels of AI, each building on the last:

**Level 1** is the ML model. XGBoost and Random Forest, trained on synthetic Australian lending data that models the same decision factors real underwriters use. 32 features, isotonic probability calibration, APRA CPG 235 drift monitoring. The synthetic data has latent variables, measurement noise, and underwriter disagreement baked in so the model produces metrics consistent with production scorecards at Australian lenders (AUC ~0.93, Gini ~0.87) rather than the 0.99 you get when your training labels are perfectly deterministic.

**Level 2** is the email system. Claude generates approval and denial letters that read like they came from a real lending officer. Every email runs through 10 deterministic guardrail checks before it goes anywhere. Hallucinated dollar amounts, discrimination act violations, AI-sounding language, missing regulatory disclosures — if any check fails, the email regenerates. Three attempts, then it flags for human review.

**Level 3** is the agentic pipeline. An orchestrator kicks off the prediction, generates the email, runs bias detection, and if it's not confident about something, it stops and escalates. Denied applicants get alternative product offers generated by a separate marketing agent with its own compliance checks. The whole flow runs as one async Celery task with step-by-step logging.

The data is synthetic. The compliance rules are real. The architecture is how you'd build this at a lender.

## The pipeline

What happens when you hit the orchestrate endpoint:

```
Application submitted
    |
    v
[1] XGBoost scores it --> probability + SHAP feature importances
    |
    v
[2] Claude writes the email --> approval or denial letter
    |
    v
[3] Guardrails check the email --> 10 deterministic checks
    |
    v
[4] Bias pre-screen (regex) --> scores 0-100
    |
    |-- score <= 60: pass, send the email
    |-- score 60-80: Claude reviews the flags
    |       \-- confidence < 0.70? escalate to human
    \-- score > 80: straight to human review
    |
    v
[5] Email sends (or gets held)
    |
    v
[6] If denied --> NBO engine generates alternative offers
    |               \-- marketing email + its own bias check
    v
[7] Status updates, frontend picks it up on next poll
```

If any step fails, the application goes to "review" and the agent run logs where it broke. Stuck pipelines auto-recover after 5 minutes — the orchestrator detects stale `processing` status and cleans up zombie runs.

## Customer experience

The system has two separate interfaces: an admin/officer dashboard and a customer-facing application portal.

### Customer application flow

1. Customer registers and logs in → lands on `/apply`
2. System checks profile completeness against NCCP Act and AML/CTF requirements
3. If incomplete, customer is directed to `/apply/profile` to fill in personal details, identity documents, employment, income, assets, liabilities, and living situation
4. Banking relationship data (tenure, balances, products, payment history) is pre-populated by the bank — customers cannot edit these
5. Once complete, customer submits a 5-step loan application (personal → employment & income → expenses & debts → loan details → review)
6. Application enters the agent pipeline (ML prediction → email generation → bias check → delivery)
7. Customer sees real-time status updates on `/apply/status/{id}`
8. Decision email is sent to the customer's email address via Gmail SMTP

### Profile security model

The edit profile page (`/apply/profile/edit`) restricts which fields customers can change online, following the same pattern as CBA, ANZ, and Westpac online banking:

**Locked (require branch visit or admin):** Name, email, date of birth, residency status, primary/secondary identity documents. These are frozen after initial submission under AML/CTF Act 2006 to prevent identity fraud. ID numbers are masked in the UI.

**Editable online:** Phone, address, marital status, contact preference, employment details, income, assets, liabilities, living situation. These are life-circumstance fields that change naturally and Australian banks allow customers to update through self-service.

**Bank-managed (read-only):** Account tenure, loyalty tier, product holdings, balances, payment history. These are derived from the banking relationship and cannot be self-reported.

### Seeded data design

The `seed_profiles` command only populates bank-known data (balances, banking relationship metrics). All customer-entered fields are left blank, so demo customers see empty forms — not pre-filled fake data. This accurately represents a new customer who has a banking relationship but hasn't completed their loan application profile.

## ML model — making synthetic data behave like real data

This is where most people building portfolio projects get it wrong. They generate clean synthetic data, train a model on it, get 99% accuracy, and call it a day. A recruiter who's worked at a bank will look at those numbers and know the data was trivial.

Real lending models at Australian banks hit AUC 0.75-0.85. Not because the models are bad, but because real lending decisions involve factors the model can't see. The underwriter reads the applicant's bank statements and notices irregular deposits. They call the employer and the phone number doesn't match. The documentation is technically complete but the tax returns don't line up with the payslips. None of that shows up in structured application data.

I wanted the model metrics to reflect that reality. Here's what I did:

**Latent variables.** The data generator computes approval decisions using factors that are never included as model features. Documentation quality (how clean the paperwork is), savings pattern quality (whether the applicant has 3+ months of consistent deposits), employer/industry stability rating, and relationship banking bonuses. These are things real underwriters assess from source documents but that don't exist in the application form. They account for roughly 40% of the composite score on borderline cases, which means the model has an irreducible error floor — it literally cannot see what drove the decision.

**Underwriter disagreement.** Two credit assessors given the same application will disagree 10-15% of the time on borderline cases. APRA's own thematic reviews document this. The generator adds normally distributed noise (sigma=0.12) to the composite score, which means some applications near the threshold get approved that "should" have been denied and vice versa. The model can't predict which way the noise went.

**Measurement noise.** The approval decision is computed on the applicant's true financial position — verified income, actual expenses, real credit score at assessment time. But the model trains on the application form data, which has:
- Income variance of +/-8% (applicants round, estimate, or report pre-tax vs post-tax inconsistently)
- Expense under-reporting of ~12% on average (well-documented in ASIC reviews of responsible lending practices)
- Credit score drift of +/-40 points between the bureau pull date and the application date

This means the model is trained on slightly wrong versions of the features that were used to make the actual decision. Exactly like production.

**Missing data.** 3-4% of optional fields (monthly expenses, credit card limits, property valuations) are null. The model has to handle this through imputation rather than relying on complete data.

**Soft policy overrides.** 5% of approvals get flipped to denials (simulating fraud flags, AML holds, verification failures that happen during processing). 5% of borderline denials get rescued by senior credit officers exercising discretion for good existing customers. These overrides are invisible to the model.

The result:

| Metric | This project | Typical AU bank | What a toy project looks like |
|--------|-------------|-----------------|-------------------------------|
| AUC-ROC | 0.93 | 0.75-0.85 | 0.99 |
| Gini | 0.87 | 0.50-0.70 | 0.98 |
| Accuracy | 0.87 | 0.80-0.88 | 0.97 |
| F1 | 0.81 | 0.75-0.85 | 0.95 |
| Brier Score | 0.09 | 0.10-0.15 | 0.02 |
| Train AUC | 0.96 | <0.95 | 1.00 |

Still slightly above real-world (the synthetic data has fewer product types, less applicant diversity, and no temporal distribution shift), but in the right ballpark. A model with these numbers is actually useful for making lending decisions. A model with 0.99 AUC is memorising its training data.

### Calibrated against real Australian data

Every distribution parameter in the generator is calibrated against published government and industry statistics — not guessed. The generator uses three statistical techniques working together:

**Gaussian copula** for correlated feature generation. In real lending data, income, credit score, employment length, expenses, and credit card limits are all correlated. Higher earners have better credit scores (r=0.26), spend more (r=0.34), and have longer employment tenure (r=0.19). The copula preserves each feature's individual distribution while introducing these realistic pairwise correlations. Without it, the model trains on artificially independent features.

**Sub-population mixture model** with five borrower archetypes. Real applicants cluster into distinct segments: first home buyers (18%, young, high LVR, moderate income), upgraders (20%, mid-career, moderate LVR), refinancers (10%, established, low LVR, excellent credit), personal borrowers (37%, auto/education/personal loans), and business borrowers (15%, older, self-employed skew). Each sub-population has its own income, credit score, loan size, and LVR distributions. This produces naturally bimodal patterns — LVR clusters around 60% and 85%, just like real bank data.

**Geographic segmentation** across all 8 Australian states and territories. A Sydney applicant borrowing $640K at 70% LVR looks very different from a Hobart applicant borrowing $468K at 85% LVR, even though both are typical in their markets. State-level parameters include income multipliers (WA 1.12x for mining, ACT 1.25x for public service, TAS 0.88x), Equifax credit score adjustments (ACT +51 from national average, NT -20), and property value scaling.

### Lending rules

The approval function implements an 11-step assessment:
- APRA 3% serviceability buffer above the product rate
- Melbourne Institute HEM benchmarks for minimum living expenses (CPI-indexed 2025/2026)
- Income shading by employment type (PAYG permanent 100%, casual 80%, self-employed 75%, contract 85%)
- Stage 3 marginal tax rates (effective 1 July 2024)
- LVR thresholds with LMI premium capitalisation (80/85/90/95% bands)
- Genuine savings requirement for high-LVR home loans
- DSR caps by income band, assessed at the APRA buffer rate
- HECS/HELP compulsory repayment at 3.5% of gross (age-correlated: 55% of under-35s, 30% of 35-55)
- Equifax Australia credit scoring scale (0-1200, national average 864)
- Composite scoring with observable + hidden (latent) components
- Soft policy overrides for fraud/AML flags and senior credit officer discretion

### Credit risk techniques

**WOE/IV scorecard.** A traditional Weight of Evidence logistic regression scorecard is built alongside the XGBoost model. WOE transforms each feature into its log odds ratio per bin; Information Value ranks features by predictive power (IV < 0.02 = drop, 0.02-0.10 = weak, 0.10-0.30 = medium, 0.30-0.50 = strong). The scorecard uses base score 600 with PDO 20 (Points to Double the Odds) — the same format used by CBA, ANZ, Westpac, and NAB. Scorecard AUC ~0.78 vs XGBoost ~0.93 — the interpretability/accuracy tradeoff that banks actually make, because APRA requires that a credit officer can explain every decision.

**Monotonic constraints.** XGBoost enforces 13 domain-consistent monotonic relationships during tree construction, including 4 on interaction features: higher credit score always increases approval probability, higher DTI always decreases it, compounding leverage (LVR x DTI) always hurts, and higher serviceability buffer always helps. Prevents the model from learning spurious patterns from noise — a regulatory expectation under APRA and ASIC guidelines.

**Feature interactions.** Four cross-feature signals standard in Big 4 scorecards: `lvr_x_dti` (compounding leverage — high LVR + high DTI is much riskier than either alone, per RBA FSR 2022), `income_credit_interaction` (joint creditworthiness and capacity), `serviceability_ratio` (monthly buffer after all commitments), and `employment_stability` (employment type quality weighted by tenure). APRA CPG 235 specifically requires interaction effects in model risk documentation.

**Conformal prediction.** Instead of just "72% probability", produces confidence intervals with guaranteed coverage: "72% [65%, 79%] at 95% confidence." Uses split conformal prediction with nonconformity scores computed on the validation set during training. Stored in the model bundle for inference-time use.

**Credit-score-sensitive LGD.** The Expected Loss model (EL = PD x LGD x EAD) adjusts Loss Given Default by credit score: better credit reduces LGD by up to 20% (borrowers cooperate with recovery, have more assets), worse credit increases it. This is standard practice at Big 4 banks under APRA APS 113.

**Reject inference.** In real lending, outcomes are only observed for approved loans — denied applicants never get a chance to prove they'd repay. The generator uses the parcelling method to estimate hypothetical outcomes for denied applicants based on their feature similarity to the approved population. Labels stored as a separate instance attribute (never in the training CSV) to prevent target leakage.

### Model metrics

The model tracks banking-standard metrics beyond the usual accuracy/F1: Gini coefficient (primary discrimination metric for AU banks), KS statistic, expected calibration error, decile lift analysis, and threshold optimisation with a banking cost function (5x penalty for false positives vs 1x for false negatives — denying a good customer is expensive but approving a bad loan is worse).

Fairness metrics check disparate impact ratio across employment type and applicant type groups against the EEOC 80% rule. Drift monitoring stores the training distribution and computes per-feature PSI on incoming applications per APRA CPG 235 requirements.

### Risk analytics

**Expected Loss (EL = PD x LGD x EAD)** — the formula that drives capital allocation at every Australian ADI under Basel III / APRA APS 113. The model outputs not just approve/deny but a dollar-value risk amount: Probability of Default (from the model) multiplied by Loss Given Default (lookup by loan purpose and LVR band: home loans 15-40%, unsecured 75%) multiplied by Exposure at Default (outstanding loan balance). This drives risk-based pricing — riskier borrowers pay higher rates.

**Stress testing** — four adverse scenarios (interest rate +2%, income -15%, property value -20%, combined) re-run each prediction under stress. Shows the model degrades sensibly: worse inputs produce lower approval probabilities. Required under APRA APS 110.

**Conformal prediction** — instead of just "72% probability", outputs confidence intervals (e.g., "72% [65%, 79%] at 95% confidence") using split conformal prediction with nonconformity scores from calibration.

### Self-healing validation

A `validate_pipeline_consistency()` method runs after every training and catches:
- Feature column mismatches between trainer and predictor
- Monotonic constraint features missing from the feature set
- Missing imputation values in the model bundle
- Target-leaking columns in training data (e.g., `_reject_inference_label`)

If any check fails, training raises a clear error listing all failures. This prevents the pipeline from silently producing incorrect predictions after code changes.

## Why this model can't be 100% accurate — and why that's the point

This section explains the deliberate design decisions and inherent limitations that prevent this model from reaching 100% accuracy. Understanding these limitations is arguably more important than the model itself — it demonstrates awareness of what ML can and cannot do in credit risk.

### What no synthetic data model can replicate

**Real loan performance data takes years to collect.** A bank needs to observe who actually defaulted over the full loan term (2-30 years) to build a PD model. Synthetic data can simulate the application decision, but not the repayment journey. The approval labels in this project are generated by rules; real labels come from watching thousands of loans play out over years. This is the fundamental gap — and it cannot be closed without working at a bank.

**Bureau depth.** This model uses a single credit score integer (Equifax 0-1200). Real bureau data includes: number of enquiries in the last 3/6/12 months, number of defaults, total credit exposure across all facilities, repayment history on each existing account, court judgments, directorship history, and cross-references to related parties. That's 50+ features from the bureau alone. A single score is a lossy compression of all that information.

**Behavioural data.** Banks with existing customers use transaction-level data: salary credit patterns, gambling transactions, savings behaviour, overdraft usage, missed direct debits, spending categories. This data is never publicly available and adds significant predictive power that no external model can access.

**Temporal dynamics.** Real lending data shifts over time: interest rate cycles, housing booms and busts, COVID hardship periods, regulatory changes (APRA buffer adjustments, responsible lending reforms). This generator produces a static snapshot. Banks retrain quarterly and track model degradation over years.

### What is deliberately approximate

**AUC of 0.93 vs real-world 0.75-0.85.** The model's AUC is higher than what a real bank achieves — not because the model is better, but because synthetic data has fewer dimensions of chaos. The latent variables, measurement noise, and underwriter disagreement create some irreducible error, but real life adds fraud, divorce, job loss, pandemics, and regulatory changes mid-loan that are impossible to simulate comprehensively.

**Geographic segmentation at state level vs postcode level.** Real banks use 2,500+ postcode-level risk ratings with suburb-specific property valuations. This project uses 8 state-level profiles — correct in direction (ACT highest income/credit, TAS lowest) but coarser in resolution.

**5 sub-populations vs 20+.** Real applicant pools include investors, SMSF borrowers, construction loan applicants, rural borrowers, non-residents, company borrowers, and trust structures. This project models 5 major segments, which captures the dominant patterns but misses the long tail of niche products.

**Income from ABS aggregates vs real tax returns.** Income distributions are calibrated against ATO and ABS published statistics, but these are population-level aggregates. A real bank sees individual tax returns with specific deductions, rental income, capital gains, overtime patterns, and bonus structures that produce a different joint distribution than any aggregate can capture.

### Design decisions and their tradeoffs

| Decision | Why | Tradeoff |
|----------|-----|----------|
| Synthetic data over real data | Privacy, legality, reproducibility | Model learns from rules, not from reality |
| XGBoost + WOE scorecard side by side | XGBoost for accuracy, WOE for interpretability | Two models to maintain; WOE AUC (~0.75) much lower than XGBoost (~0.93) |
| Monotonic constraints (9 features) | Regulatory expectation; prevents spurious patterns | Reduces model flexibility; XGBoost can't capture certain interactions |
| Gaussian copula for correlations | Features are correlated in reality | Copula captures pairwise correlations but not higher-order dependencies |
| APRA assessment rate as stress test, not hard cutoff | Real banks approve loans that pass at product rate even if tight at assessment rate | More realistic approval rates (66% vs 30% without this) |
| Reject inference (parcelling) | Corrects selection bias from only observing approved outcomes | Simplest method; banks use more sophisticated variants (Heckman, augmentation) |
| Geographic at state level | Captures major income/credit/property variation | Misses within-state variation (Sydney CBD vs western Sydney vs regional NSW) |

### What would be needed for production deployment

This system demonstrates the architecture, methodology, and regulatory awareness needed for a real lending platform. To deploy at an actual bank, you would additionally need:

1. **Real loan performance data** — minimum 3-5 years of repayment history across economic cycles
2. **Full bureau integration** — Equifax/illion/Experian API with comprehensive credit file access
3. **Model validation by an independent team** — APRA requires that the team validating the model is separate from the team that built it
4. **Champion/challenger testing** — run the new model alongside the existing model for 6-12 months before switching
5. **Stress testing under APRA scenarios** — specific macroeconomic scenarios provided by the regulator
6. **Model documentation per APRA CPG 235** — formal model risk management documentation
7. **Ongoing monitoring infrastructure** — automated PSI alerts, quarterly model reviews, annual revalidation

The gap between this project and production is data and process, not methodology. The statistical techniques (WOE, copula, calibration, monotonic constraints, PSI, SHAP) are the same ones used by CBA, ANZ, Westpac, and NAB.

## Email guardrails

Every email Claude generates goes through these before it can send:

1. **Prohibited language** — regex patterns derived from the Sex Discrimination Act, Racial Discrimination Act, Disability Discrimination Act, and Age Discrimination Act. Legal disclosure text is stripped before checking so references to the acts themselves don't trigger false positives.
2. **Hallucinated numbers** — every dollar amount in the email gets validated against the application data and pricing engine. ASIC comparison rate footnotes (which contain standard regulatory amounts like $30,000) are stripped before checking.
3. **Tone** — flags aggressive language ("demand", "threat", "unacceptable")
4. **AI giveaway language** — 30+ patterns. "Delighted", "rest assured", "every step of the way", hedging qualifiers, performative empathy, transitional adverbs. If it reads like an LLM wrote it, it fails.
5. **Professional financial language** — no "guaranteed approval", no "risk-free", no "exclusive for you"
6. **Plain text only** — rejects markdown, HTML, em dashes. Allows Unicode bullets and en dashes.
7. **Word count** — approval: 650 max, denial: 500 max
8. **Required elements** — approval emails need next steps, cooling-off period, hardship provisions, AFCA reference. Denial emails need reasons (Banking Code para 81), free credit report rights, dispute resolution info.
9. **Sign-off structure** — catches double closings (a common LLM failure mode)
10. **Sentence rhythm** — flags suspiciously uniform sentence lengths (std dev < 3.0). AI text tends to produce every sentence at ~15 words. Warning only — provides retry feedback but doesn't block sending.

If any blocking check fails, the email regenerates with the failure details appended to the prompt. Three attempts max, then human review.

## Bias detection

The system doesn't throw the email at an LLM and ask "is this biased?" That approach gives you hallucinated bias flags and false confidence. Instead it runs deterministic checks first and only calls the LLM when it needs interpretation.

**Layer 1: Regex pre-screen.** Runs on every email. No API cost. Catches prohibited terms, tone violations, discriminatory framing. Each violation type adds weighted points. Score 0-100.

**Layer 2: LLM review.** Only fires if the score is between 60 and 80 (moderate flag zone). Claude Sonnet examines the specific flags and determines if they're false positives. Legal disclosures and standard banking terms trigger regex patterns all the time but aren't actually biased. This layer sorts that out.

**Layer 3: Senior compliance.** Only fires if Layer 2 approves with low confidence (<0.70). Escalates to human review. No email goes out without confidence from at least one review layer.

Marketing emails use tighter thresholds (pass: 50, review: 70) because there's more room for subtle bias in sales copy aimed at declined applicants.

## Security

JWT auth with 60-min access tokens, 7-day refresh, rotation, and blacklisting. Passwords hashed with Argon2. PII fields encrypted with Fernet. Rate limiting at 20/min anonymous, 60/min authenticated, 60/hour on the pipeline endpoint. CORS locked to the frontend origin. X-Frame-Options DENY. Three roles: admin, officer, customer with permission checks on every endpoint. Async task results stored in PostgreSQL so they survive broker restarts. Prompt injection defences on all user-supplied text that enters LLM prompts.
