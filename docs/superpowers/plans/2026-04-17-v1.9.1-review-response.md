## Task A: DataGenerator post-outcome leak regression test

**Why first:** Correctness item. Touches no CI or workflow files. Zero dependency on other tasks.

**Background for the engineer:** `backend/apps/ml_engine/services/trainer.py` defines `NUMERIC_COLS` (lines 90–208) and `CATEGORICAL_COLS` — these are the features that actually reach the XGBoost model. `DataGenerator.generate()` (at `data_generator.py:466`) emits many columns, some of which represent information only available AFTER a lending decision (e.g., the "computed after approval" section at `data_generator.py:1259`). If any of those post-outcome columns are ever added to `NUMERIC_COLS`/`CATEGORICAL_COLS`, the model would train on target-leakage features — a methodology bug.

**Audit finding (from 2026-04-17 brainstorming):** current training set does NOT include any post-outcome features — but the separation is informal. This task promotes it into a code-enforced invariant.

- Create: `backend/tests/test_data_generator_no_leak.py`

- [ ] **Step 2: Write the failing regression test**

Create `backend/tests/test_data_generator_no_leak.py` with this exact content:

```python
"""Regression test: no post-outcome feature may reach the training feature set.

A post-outcome feature is one whose value is only knowable AFTER a lending
decision has been made (e.g., realised default flags, prepayment buffer
computed after approval, ex-post loan performance metrics). Including any
such feature in training is target leakage — it inflates validation AUC
without generalising to real prediction-time data.

This test enforces the separation as a code-level invariant so the pattern
cannot be reintroduced without a deliberate test change.
"""

from apps.ml_engine.services.data_generator import POST_OUTCOME_FEATURES
from apps.ml_engine.services.trainer import ModelTrainer


def test_training_features_exclude_post_outcome_columns():
    training_features = set(ModelTrainer.NUMERIC_COLS) | set(ModelTrainer.CATEGORICAL_COLS)
    leaked = training_features & POST_OUTCOME_FEATURES
    assert not leaked, (
        f"Post-outcome features leaked into training set: {sorted(leaked)}. "
        f"These columns represent information only available AFTER a lending "
        f"decision and must not be used as model inputs."
    )


def test_post_outcome_features_constant_is_not_empty():
    """Sanity check: if this constant became empty, a future refactor silently
    defanged the leakage test above. Fail loudly instead."""
    assert len(POST_OUTCOME_FEATURES) >= 3, (
        f"POST_OUTCOME_FEATURES has only {len(POST_OUTCOME_FEATURES)} entries. "
        f"Expected at least 3 known post-decision columns from DataGenerator."
    )
```

## Notes for the engineer

- **TDD discipline**: for every test you add (Tasks A and B especially), write the test first, watch it fail, write the minimum code, watch it pass. Do not skip the red-fail step — it's what proves the test actually exercises the code.
- **One commit per logical change**: never stack "fix typo" and "add new feature" in one commit. Split them.
- **Pre-commit hook stack** is enforced locally (ruff, bandit, pip-audit, gitleaks). If a hook fails, fix the underlying issue — never `--no-verify`.
- **Push at end of each task**: don't bundle a day's work into one push. Small frequent pushes mean CI catches issues early.
- **When a step says "expected: FAIL"**, take the failure message seriously. A confusing failure message is a signal that the test is badly written, not that you should hack around it.
- **If CI is flaky**, do not paper over it with retries. The v1.9.0 polish pass already found and fixed a real latent production bug (Postgres nullable-OneToOne locking) by treating a test flake as a symptom. Treat the next flake the same way.
