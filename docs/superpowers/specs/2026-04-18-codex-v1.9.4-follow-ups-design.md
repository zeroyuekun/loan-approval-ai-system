## Finding 2 — Batch orchestration hard cap

**Problem.** `BatchOrchestrateView.post()` defines `max_batch = 100` as "a safety cap to prevent accidental mass re-processing," but the cap is only applied in the `recheck=true` branch. The default branch materializes every pending application via `.values_list("id", flat=True)` and dispatches one Celery task per row with no bound. A single staff click can therefore dump the entire backlog into the broker, causing queue spikes, duplicate expensive work, and hard-to-recover operational incidents.

**Chosen approach.** Apply the same cap to both branches. Expose the skipped count in the response and audit entry so operators know to drain the backlog in batches rather than silently losing visibility.

**Design.**

Edit `backend/apps/agents/views.py`:

- Lift the cap to a module-level constant so tests can import it:
  ```python
  BATCH_ORCHESTRATE_MAX = 100
  ```
- Each branch computes its own `total_eligible` count *before* slicing so the skipped count is accurate regardless of status transitions that happen during the recheck loop:
  ```python
  def post(self, request):
      recheck = request.query_params.get("recheck", "").lower() == "true"

      if recheck:
          reviewable_qs = (
              LoanApplication.objects
              .filter(status__in=[
                  LoanApplication.Status.REVIEW,
                  LoanApplication.Status.PROCESSING,
              ])
              .exclude(status=LoanApplication.Status.PROCESSING)
              .order_by("created_at")
          )
          total_eligible = reviewable_qs.count()
          candidates = reviewable_qs[:BATCH_ORCHESTRATE_MAX]
          pending_ids = []
          for app in candidates:
              try:
                  app.transition_to(
                      "pending",
                      user=request.user,
                      details={"reason": "batch_recheck"},
                  )
                  pending_ids.append(app.id)
              except LoanApplication.InvalidStateTransition:
                  continue
      else:
          pending_qs = (
              LoanApplication.objects
              .filter(status=LoanApplication.Status.PENDING)
              .order_by("created_at")
          )
          total_eligible = pending_qs.count()
          pending_ids = list(
              pending_qs.values_list("id", flat=True)[:BATCH_ORCHESTRATE_MAX]
          )

      skipped = max(0, total_eligible - len(pending_ids))
      # ...dispatch loop, audit log with skipped_count, response...
  ```
  - `total_eligible` for recheck is the count of reviewable-but-not-currently-processing rows at the start of the call. Rows that later fail `InvalidStateTransition` during the transition loop are counted as dispatched-but-failed internally, not as `skipped`. That is a pre-existing failure mode unrelated to the cap.
  - `total_eligible` for the default path is the count of `PENDING` rows at query time.
- Response shape on the happy path (no overflow) stays byte-identical to today: `{"queued": N, "tasks": [...]}`.
- On overflow, add two keys:
  ```json
  {
    "queued": 100,
    "skipped": 42,
    "detail": "42 more pending; call again to continue",
    "tasks": [...]
  }
  ```
- Audit log `details` dict gains `skipped_count` (always present, integer, zero on happy path).
- Ordering becomes explicit oldest-first (`.order_by("created_at")`) so the drain is fair — today's implicit ordering is FK-insertion which happens to match but is not guaranteed.

**Alternative considered.** Cursor pagination (`?cursor=<id>`). Rejected — the backlog drain is a rare operational path, not a user-facing list, and the extra surface area invites bugs. "Call again" is a sufficient contract for the one staff user who runs it.

**Alternative considered.** Per-request `?batch_size=N` override bounded by a hard ceiling. Rejected — gives staff one more way to accidentally flood the broker for no real benefit.

**Tests.**

New file `backend/tests/test_batch_orchestrate_cap.py`:

- default path with 150 pending apps, mock `orchestrate_pipeline_task.delay` → response has `queued=100`, `skipped=50`, `detail` contains "50 more pending"; `.delay` called exactly 100 times
- default path with 30 pending apps → response is `{"queued": 30, "tasks": [...]}` with no `skipped`/`detail` keys (happy-path regression guard)
- recheck path with 150 reviewable apps → `queued=100`, `skipped=50`; unreachable rows remain in their original status (not transitioned)
- ordering verified: seed five apps with descending `created_at`, assert dispatch order is oldest-first
- `AuditLog` row exists with `details["skipped_count"] == 50` on overflow and `== 0` on happy path
- empty case (no pending) → existing `{"detail": "No pending applications to process.", "queued": 0}` unchanged

## Finding 3 — Complaint ownership validation

**Problem.** `ComplaintSerializer.create()` overwrites `complainant` with `request.user` but never validates that the referenced `loan_application` belongs to that user. `ComplaintViewSet` allows any authenticated user to create complaints, so a customer can file a complaint tied to another customer's application ID. This breaks tenant isolation, pollutes the complaint workflow with forged links, and leaks application-id existence via the validation response.

**Chosen approach.** Add a `validate_loan_application` method on the serializer. Staff (admin/officer) can reference any loan (they legitimately file complaints on behalf of customers per ASIC RG 271 handling). Non-staff can only reference loans where `loan_application.applicant == request.user`. Anonymous complaints (`loan_application is None`) remain supported.

**Design.**

Edit `backend/apps/loans/serializers.py`:

```python
def validate_loan_application(self, value):
    if value is None:
        return value  # Anonymous / non-loan-linked complaints allowed
    request = self.context["request"]
    user = request.user
    if getattr(user, "role", None) in ("admin", "officer"):
        return value
    if value.applicant_id != user.id:
        raise serializers.ValidationError(
            "You can only file complaints on your own applications."
        )
    return value

def create(self, validated_data):
    from datetime import timedelta
    from django.utils import timezone
    from apps.accounts.models import AuditLog

    now = timezone.now()
    request = self.context["request"]
    user = request.user
    loan_app = validated_data.get("loan_application")

    validated_data["complainant"] = user
    validated_data["acknowledged_at"] = now

    category = validated_data.get("category", "other")
    sla_days = 21 if category == "decision" else 30
    validated_data["sla_deadline"] = now + timedelta(days=sla_days)

    instance = super().create(validated_data)

    on_behalf_of_id = None
    if loan_app and loan_app.applicant_id != user.id:
        on_behalf_of_id = loan_app.applicant_id

    AuditLog.objects.create(
        user=user,
        action="complaint_filed",
        resource_type="Complaint",
        resource_id=str(instance.id),
        details={
            "category": category,
            "loan_application_id": loan_app.id if loan_app else None,
            "on_behalf_of_id": on_behalf_of_id,
        },
        ip_address=request.META.get("REMOTE_ADDR"),
    )

    return instance
```

**Alternative considered.** Check ownership in `ComplaintViewSet.perform_create`. Same effect, but DRF convention places field-level invariants on the serializer where they show up in schema introspection and OpenAPI docs. Slight preference for the serializer location.

**Alternative considered.** Drop `loan_application` from writable fields entirely; derive it from a nested URL (`/api/v1/loans/<id>/complaints/`). Rejected — the current flat endpoint is used by the dashboard and changing the API shape is disproportionate to closing the finding.

**Staff-leak consideration.** The staff override cannot reveal existence of other applications to non-staff, because a non-staff caller hits the `applicant_id != user.id` branch before touching any staff path. Staff already have full read access via existing viewsets, so the "leak" is in-role.

**Tests.**

New file `backend/tests/test_complaint_tenant.py`:

- customer A posts complaint with `loan_application=<B's loan id>` → 400, body contains "You can only file complaints on your own applications."; no `Complaint` row created; no `AuditLog` row
- customer A posts complaint with `loan_application=<A's own loan>` → 201; `AuditLog` with `details["on_behalf_of_id"] is None`
- officer posts complaint with `loan_application=<B's loan>` → 201; `AuditLog` with `details["on_behalf_of_id"] == <B.id>`
- admin same as officer (role check matches both)
- complaint with `loan_application=None` (anonymous / general) → 201; `AuditLog` with `loan_application_id=None`
- non-existent loan id → 400 (existing FK validation, regression guard)
