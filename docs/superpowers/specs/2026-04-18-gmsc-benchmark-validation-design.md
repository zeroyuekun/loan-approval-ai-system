**Date:** 2026-04-18
**Author:** Claude (with Neville Zeng)
**Status:** Approved — proceeding to implementation plan
**Target rating lever:** 9.6 → 9.7+ (portion of the 9.9 push)

## Problem

The AussieLoanAI XGBoost credit decision model is trained on synthetic data generated by `DataGenerator`, calibrated against official Australian statistical sources (ATO, ABS, APRA, Equifax, RBA, Cotality). It reports AUC 0.88 on its own synthetic test set and 0.85 on the reproducible 2k synthetic subset benchmark.

A fair critique from any reviewer: *"Does this pipeline actually work on real borrower data, or is it just fitting a synthetic distribution?"*

This spec addresses that critique with a scientifically honest **external benchmark study** against the public Kaggle "Give Me Some Credit" (GMSC) dataset — real anonymised borrower data used in a 2011 Kaggle competition (150,000 training examples, target `SeriousDlqin2yrs` = 90+ days delinquent within 2 years).

## Goal

Demonstrate that the *same XGBoost + Optuna + isotonic-calibration pipeline* used in production, when retrained on GMSC's 10 features and 150k real borrowers, reaches AUC competitive with published mid-tier leaderboard results (0.82–0.87, top-1% ≈ 0.869).

**Success criterion:** 5-fold stratified CV AUC ≥ 0.82 on GMSC, published in `MODEL_CARD.md` with full methodology.

**Acceptable honesty outcome:** If CV AUC lands outside 0.82–0.87, the result is still published with an honest discussion of why (e.g., feature engineering specific to Australian data that doesn't transfer). Concealing a low number would be a portfolio negative; publishing an honest one is a positive.

## Non-Goals

1. **Direct application of the production model to GMSC** — only ~6 of the 71 production features overlap (income, DTI, dependants, utilisation, open accounts, age). Applying the production model to GMSC would require imputing 65 Australian-specific features from dataset-wide means, which is scientifically noisy and invites challenge. Explicitly out of scope.
2. **Retraining the production model** — this benchmark is a sibling artifact. Zero changes to `predictor.py`, `trainer.py`, or the active `ModelVersion`.
3. **CI integration** — GMSC training takes minutes, not seconds. Manual invocation via `make benchmark-gmsc` or `python scripts/benchmark_gmsc.py`. A smoke test (CSV load + feature mapping) is CI-safe; the full benchmark is not.
4. **Hyperparameter comparison across Optuna vs default** — single run with same Optuna budget as production (50 trials) is sufficient. A full comparison table is scope-creep.
5. **Fairness / subgroup analysis on GMSC** — GMSC does not publish protected-attribute data. Fairness analysis remains in the synthetic pipeline where it belongs.

## Approach

### Data Acquisition

- **Source:** Public GitHub mirror `https://github.com/DrIanGregory/Kaggle-GiveMeSomeCredit` (contains `cs-training.csv`, 150,000 rows).
- **Why not Kaggle API:** Avoids requiring reviewers to register a Kaggle account + API token. Mirror is read-only and stable (last updated 2019).
- **Cache location:** `.tmp/gmsc/cs-training.csv`. Never committed (already in `.gitignore` via `.tmp/`).
- **License:** Original Kaggle competition rules apply (competition data, research/educational use). Explicitly noted in `docs/experiments/gmsc_benchmark.md`. No data redistributed by this repository.
- **Integrity check:** SHA256 of downloaded file asserted against a pinned hash in the benchmark script. If the upstream mirror changes, the run fails loudly rather than silently benchmarking on different data.

### Feature Schema (GMSC)

| Column | Type | Description | Maps to |
|---|---|---|---|
| `SeriousDlqin2yrs` | bool | **Target**: 90+ days delinquent within 2 years | n/a (label) |
| `RevolvingUtilizationOfUnsecuredLines` | float | Total balance / credit limit | ~ `credit_utilization_pct` |
| `age` | int | Borrower age | derived in production from DOB |
| `NumberOfTime30-59DaysPastDueNotWorse` | int | 30-59 day delinquency count | ~ `num_late_payments_24m` (partial) |
| `DebtRatio` | float | Monthly debts / monthly income | ~ `debt_to_income` |
| `MonthlyIncome` | float | Monthly gross income | ~ `annual_income / 12` |
| `NumberOfOpenCreditLinesAndLoans` | int | Open credit lines and loans | ~ `total_open_accounts` |
| `NumberOfTimes90DaysLate` | int | 90+ day delinquency count | ~ `worst_late_payment_days` (partial) |
| `NumberRealEstateLoansOrLines` | int | Real-estate loans / lines | no direct production field |
| `NumberOfTime60-89DaysPastDueNotWorse` | int | 60-89 day delinquency count | no direct production field |
| `NumberOfDependents` | int | Dependants | ~ `number_of_dependants` |

~6 meaningful overlaps. Documented in `docs/experiments/gmsc_benchmark.md` with a "What this DOES NOT validate" subsection listing the 65 non-overlapping production features.

### Pipeline

Standalone script at `backend/scripts/benchmark_gmsc.py`:

1. Ensure data cached → download + SHA256 verify if missing
2. Load CSV → pandas DataFrame (150k rows, 11 cols)
3. Preprocess:
   - Drop `MonthlyIncome` NaNs via median imputation (well-known GMSC preprocessing step — ~20% missing)
   - Drop `NumberOfDependents` NaNs via zero imputation
   - No synthetic SMOTE or oversampling — report performance on the natural imbalance (~6.7% positive class)
   - Cap outliers at 99th percentile for `DebtRatio` and `MonthlyIncome` (standard GMSC treatment)
4. Train `XGBoostClassifier` with Optuna hyperparameter search (50 trials, same space as production: `max_depth ∈ [3, 10]`, `learning_rate ∈ [0.01, 0.3]`, `n_estimators ∈ [100, 1000]`, `subsample ∈ [0.6, 1.0]`)
5. Wrap in scikit-learn `CalibratedClassifierCV` (isotonic, 3-fold) — matches production
6. 5-fold stratified CV → report mean ± std AUC, KS statistic, Brier score
7. Output JSON: `.tmp/gmsc/benchmark_results.json` (not committed)

### Honesty gates

- Script fails loudly (non-zero exit) if:
  - SHA256 mismatch
  - CV AUC std > 0.02 (instability signal)
  - Dataset row count != 150,000 (wrong file)
- Results always published regardless of outcome. No "retry until good."

### Reproducibility

Script is idempotent:
- Cached data → skip download
- Seeded splits (`random_state=42`) → identical AUC on re-run
- Emits `python --version`, `xgboost.__version__`, `optuna.__version__`, `sklearn.__version__` into results JSON

## Testing Strategy

| Test | Type | CI? |
|---|---|---|
| Feature mapping produces expected column subset | Unit (mock CSV, 10 rows) | ✅ |
| SHA256 mismatch raises | Unit | ✅ |
| Median imputation for `MonthlyIncome` NaNs | Unit | ✅ |
| Full benchmark run | Integration | ❌ (manual) |
| AUC within 0.82–0.87 expected range | Integration assertion (soft — warn, not fail) | ❌ |

Unit tests total ≤200 lines, no heavy deps. No network calls in CI.

## Risks & Mitigations

| Risk | Likelihood | Mitigation |
|---|---|---|
| Upstream GitHub mirror goes offline | Low | SHA256 check + fallback to Kaggle API path (documented but not auto-run) |
| AUC lands below 0.82 | Low-Med | Publish honestly; add a discussion section. Does not block merge. |
| Windows + SIGALRM incompatibility (script hangs) | Low | Optuna has its own time budgets; no SIGALRM use in benchmark script |
| XGBoost version drift changes numerics | Low | Version pinning in results JSON; minor AUC drift is expected and acceptable |
| User doesn't want `.tmp/gmsc/` downloaded | Low | Script prompts "download 7MB from GitHub? [y/N]" if TTY; `--yes` flag for CI-like runs |

## Rollback

Pure additive change. To roll back: `git revert <pr-sha>`. No migrations, no production model changes, no schema changes, no config changes. Zero production impact.
