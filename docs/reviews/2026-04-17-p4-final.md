## Finding detail

### G-02 · `select_for_update` against nullable OneToOne join

**File:** `backend/apps/agents/services/human_review_handler.py:26-38`

**What it is:** the resume-pipeline handler opens its transaction with

```python
AgentRun.objects.select_for_update().select_related(
    "application__applicant",
    "application__decision",   # nullable OneToOne
).get(pk=agent_run_id)
```

The `application.decision` relation is a nullable OneToOne (a `LoanDecision` is created partway through the pipeline). Combining a nullable OneToOne in `select_related` with `select_for_update` produces a `LEFT OUTER JOIN ... FOR UPDATE`, which Postgres rejects with `NotSupportedError: FOR UPDATE cannot be applied to the nullable side of an outer join`.

**Why it matters:** the bug is latent on SQLite (dev) but blows up on the first human-review resume in Postgres (CI + prod). It's only visible when a real Postgres instance actually executes the query. The B5 assertion pattern (`isinstance(result.result, AgentRun.DoesNotExist)`) surfaced it because the stronger expectation failed — the task didn't raise `AgentRun.DoesNotExist`, it raised the Postgres error instead.

**Fix applied (inside PR #46):** dropped `application__decision` from the `select_related` inside the locked query. Decision is accessed separately after the lock block (Django issues a follow-up SELECT without LEFT JOIN). `applicant` stays in `select_related` because the FK is non-nullable. Test assertion now passes on Postgres.
