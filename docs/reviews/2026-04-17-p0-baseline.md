## Finding details

### F-01 · Orchestrator bypasses `transition_to()` — no `AuditLog` for ML pipeline final decisions

**Category:** compliance / data-integrity
**Where:** `backend/apps/agents/services/orchestrator.py:347,363,424,467`
**What:** All four final-decision status transitions use the raw ORM pattern
`LoanApplication.objects.filter(pk=application.pk).update(status=...)` inside
a `transaction.atomic()` block. This bypasses `LoanApplication.transition_to()`
(in `backend/apps/loans/models.py:318-343`), which is the single source of
truth for (a) validating the transition is legal via `ALLOWED_TRANSITIONS`
and (b) writing an `AuditLog(action="status_transition")` row.
**Why it matters:** The ML pipeline is the single largest source of final status
transitions in the system. Under NCCP Act s.136 and ASIC RG 209 a lender must
maintain an auditable trail for every credit decision. The current code
produces zero `AuditLog` entries for final ML-pipeline decisions — the audit
trail has a gap precisely where regulators would look first.
**Suggested fix:** Replace the raw `.update()` with a `refresh_from_db()` +
`application.transition_to(...)` call inside the same atomic block. Same
pattern applies to F-02 and F-03.

### F-02 · Email pipeline bypasses `transition_to()` for bias-triggered REVIEW escalations

**Category:** compliance / data-integrity
**Where:** `backend/apps/agents/services/email_pipeline.py:222`
**What:** Same raw `.update(status=REVIEW)` pattern when bias detection
escalates an application to human review.
**Why it matters:** Bias-triggered REVIEW transitions are the highest-sensitivity
events in the pipeline; having no audit-log entry at this point removes the
evidence trail that the bias detector actually fired. Same fix as F-01.

### F-03 · Human review handler bypasses `transition_to()` for officer decisions

**Category:** compliance / data-integrity
**Where:** `backend/apps/agents/services/human_review_handler.py:207`
**What:** Same raw `.update(status=decision)` pattern when a human reviewer
approves or denies an application held for review.
**Why it matters:** The human-review resume path is the only place where a
loan officer's decision is recorded in code. No `AuditLog` means the officer's
decision leaves no audit trail — severe gap for any Responsible-Lending review
that reaches a human. Same fix as F-01.

### F-04 · `_REDIS_FALLBACK_CALLS` never resets — worker permanently blocked after Redis recovery

**Category:** latent-bug / concurrency
**Where:** `backend/apps/agents/services/api_budget.py:70-71,132-134`
**What:** The module-level counter `_REDIS_FALLBACK_CALLS` is only ever
incremented (line 133). Once it exceeds `_REDIS_FALLBACK_LIMIT = 20` during a
Redis outage, every subsequent `check_budget()` call in that worker process
raises `BudgetExhausted` — even after Redis has fully recovered. Also: the
increment is not thread-safe, and Celery runs IO workers with concurrency > 1.
**Why it matters:** A brief Redis blip (>20 calls within it) silently bricks
that worker's Claude API access until manual restart. In a portfolio demo this
turns a transient outage into a permanent outage with no alert.
**Suggested fix:** Guard the counter with a `threading.Lock`; reset it to 0 at
the top of `check_budget()` the moment `_get_redis()` succeeds.

### F-05 · Celery integration tests assert only `result is not None`

**Category:** test-weakness
**Where:** `backend/tests/test_celery_integration.py:42,51,61,70,84,97`
**What:** Six "serialisation" tests each end with `assert result is not None`.
Because the tasks are deliberately invoked against non-existent DB rows
(`id=999`), the tasks always fail internally; the EagerResult object still
exists, so the assertion passes whether serialisation worked or not. Tests
pass even if the task module fails to import.
**Why it matters:** These tests are load-bearing in CI for the claim that
the broker path is exercised. They provide false confidence — a genuine
serialisation break would not fail the build.
**Suggested fix:** Assert `result.failed() is True` and that
`result.result` is a recognisable domain exception
(`LoanApplication.DoesNotExist`, etc.). That demonstrates the task actually
executed through the broker path and failed at the expected point.

### F-06 · `dice-ml>=0.11` unpinned

**Category:** config
**Where:** `backend/requirements.txt:16`
**What:** The only dependency with a range spec (`>=`) rather than an exact
pin. DiCE has shipped breaking API changes between minor versions.
**Suggested fix:** Pin to the currently installed version.

### F-07 · `on_time_payment_pct` has no 0-100 validator

**Category:** missing-invariant
**Where:** `backend/apps/accounts/models.py:228`
**What:** `FloatField(default=100.0)` with no `MinValueValidator(0)` /
`MaxValueValidator(100)`. The sibling field `number_of_dependants` (line 209)
correctly uses both validators. The value feeds directly into RF/XGBoost.
**Suggested fix:** Add the two validators plus a data migration.

### F-08 · `celery.py` silent default to `development` settings

**Category:** config
**Where:** `backend/config/celery.py:8`
**What:** `os.environ.setdefault("DJANGO_SETTINGS_MODULE",
"config.settings.development")` — if a prod deployment forgets the env var,
workers silently run with `DEBUG=True`, `ALLOWED_HOSTS=["*"]`.
**Suggested fix:** Emit a loud `warnings.warn(...)` when the env var is absent
before the `setdefault` fires.

### F-09 · Grafana ships with `changeme` default password

**Category:** security
**Where:** `docker-compose.yml:300`
**What:** `GF_SECURITY_ADMIN_PASSWORD=${GRAFANA_ADMIN_PASSWORD:-changeme}`.
Only reachable via the `monitoring` profile, but still a default that ships.
**Suggested fix:** Switch `:-changeme` to `:?GRAFANA_ADMIN_PASSWORD must be set`
so Compose fails fast if the env var is absent.

### F-10 · Email availability probe bypasses `guarded_api_call`

**Category:** observability
**Where:** `backend/apps/email_engine/services/email_generator.py:466-484`
**What:** The 1-token Anthropic probe at line 479 calls
`self.client.messages.create(...)` directly, skipping
`ApiBudgetGuard.record_call()`. Dollar impact is negligible
(~$3×10⁻⁶ per probe) but architecturally inconsistent.
**Suggested fix:** Route the probe through `guarded_api_call`.

### F-11 · CI test Fernet key inlined in `ci.yml`

### F-12 · Bandit `--severity-level high` only

## Disposition rules
- **Fold-in** — must be fixed as part of this polish pass. Criteria:

  weakness that could hide regressions during this pass.
- **Follow-up** — meaningful but scoped work that would expand the pass.
  File as a GitHub issue.
- **Parking-lot** — style preference, micro-optimisation, or already known.
